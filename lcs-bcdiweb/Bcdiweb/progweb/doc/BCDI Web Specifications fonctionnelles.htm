1.       Présentation générale

Nota : dans cette documentation, lorsque nous parlerons d« installation standard », comprenez quil sagit des paramètres de linstallation de BCDI Web telle quelle est livrée, cest à dire des paramètres que nous avons définis à ne pas confondre avec les paramètres « par défaut » qui désignent ce qui est prévu (pré-défini) dans BCDI WEB si lutilisateur ne définit pas lui-même des paramètres.

1.       Objectif

Ce module BCDI web vient compléter le logiciel BCDI réseau client-serveur. Il permet de consulter par interface WEB  Internet ou Intranet - la base de données documentaires gérée par BCDI

2.       Contenu de lensemble

BCDI Web livré contient :

Un exécutable (CGI)  BCDIWEB.CGI pour Windows (serveur IIS) ou pour Linux (serveur Apache). Dans cette documentation, le CGI sera appelé « lapplication ».

Un modèle de fichier de configuration BcdiWeb_init.ini. Ce fichier devra être dupliqué pour créer votre propre fichier de configuration et adapté (un outil complémentaire, ParamBcdiWeb, livré avec BCDI Web, vous simplifiera cette opération..

Des fichiers HTM qui serviront de modèles de pages de formulaire et dédition des résultats. Ces fichiers pourront éventuellement (avec prudence) être modifiés pour être personnalisés.

Un journal des recherches au format Csv est créé (extension .Log).

3.      Installation

Reportez-vous à la documentation dinstallation en ligne.

4.      Le lancement de lapplication BcdiWeb

Au lancement de BCDI Web à partir de votre navigateur :

-           vous invoquez directement lapplication (lien dans votre page daccueil)
http://NomDeMonServeur/AliasBcdiWeb/Bcdiweb.cgi/demo

vous arrivez directement sur lécran de recherche correspondant au fichier Index.htm du sous-répertoire Modeles (installation standard \BcdiServ\ProgWeb\modeles, ou /usr/BcdiServ/ProgWeb/Modeles).

2.      Fichier(s) de configuration BcdiWeb_Xxxx.ini (fichier modèle  BcdiWeb_Init.ini)

1.       Structure

A partir du fichier INI modèle (Bcdiweb_Init.ini), vous pouvez créer autant de fichiers Ini que vous avez de bases à mettre en ligne en recopiant puis renommant ce fichier.
A titre dexemple, BcdiWeb_Init.ini vous est proposé, qui conduit dans la base \BcdiServ\Data, mais vous pouvez créer BcdiWeb_Demo.ini, BcdiWeb_Base1.ini, BcdiWeb_Base2.ini, etc, et ces bases seront alors accessibles depuis un navigateur par les URL
http://MonServeur/MonAlias/BcdiWeb.cgi/Demo
http://MonServeur/MonAlias/BcdiWeb.cgi/Data
http://MonServeur/MonAlias/BcdiWeb.cgi/Base1
http://MonServeur/MonAlias/BcdiWeb.cgi/Base2
etc.

Nota : Lutilisation de loutil ParamBcdiWeb.exe (livré et installé en même temps que Bcdi Web) simplifie grandement la création et la maintenance de ces fichiers INI, même sil reste possible de les gérer à la main avec un éditeur de textes.

Le fichier INI a la structure classique des fichiers de configuration : il est décomposé en sections, dont le nom est entre crochets, elles-mêmes formées de lignes du type NOM=Valeur.

Les sections peuvent être écrites en minuscules ou majuscules, sans accents. Dans les lignes, les noms peuvent être écrits en minuscules ou majuscules, sans accents. Sous Linux, attention au strict respect de la casse du chemin et du nom des fichiers.

Une ligne commençant par le signe point-virgule ( ;) est considérée comme un commentaire et nest donc pas prise en compte.

2.          Section [REPERTOIRES]

1.      DONNEES

La ligne DONNEES indique le répertoire dans lequel doit se trouver la base de données BCDI (les fichiers *.DAT et *.IDX). Dans linstallation « standard », le chemin proposé est   c:\BcdiServ\Data pour Windows et /usr/BcdiServ/Data pour Linux.

2.      TRAVAIL

La ligne TRAVAIL indique le répertoire dans lequel lapplication pourra écrire ses fichiers temporaires (fichiers *.ECS) et le journal des recherche (BcdiWeb_Demo.log, BcdiWeb_Data.log, etc). Dans linstallation « standard », il sagira de c:\BcdiServ\ProgWeb\TravInt ou /usr/BcdiServ/ProgWeb/TravInt..

3.      MODÈLES

La ligne MODELES indique le répertoire dans lequel doivent se trouver les fichiers modèles HTM de formulaire et de réponses. Dans linstallation « standard », ce répertoire sappelle SPECIAL (trois interfaces vous sont proposées COLLEGE / LYCEE et SPECIAL.
Attention ! ce répertoire DOIT être un sous répertoire du répertoire de lapplication.

4.      CACHE

Ce répertoire (par défaut CACHE=c:\BcdiServ\ProgWeb\Cache) permet de recopier les fichiers locaux (dont ladresse en fichier Ressource est locale, par exemple ..\data\obj\MonImage.jpg ou d:\Bcdiserv\data\Obj\MonTexte.pdf) pour permettre leur affichage dans BCDI web depuis un navigateur.

Attention !
Si vous modifiez le nom de ce répertoire, par exemple pour créer un répertoire cache spécifique à chaque base si vous installez plusieurs bases (ce répertoire cache doit OBLIGATOIREMENT avoir été créé par vous-même auparavant et se trouver en sous-répertoire du répertoire dinstallation de Bcdi web), vous devez AUSSI créer une ligne supplémentaire dans la section [MODELES] avec la syntaxe suivante :
CACHE=chemin html depuis le répertoire dinstallation de BCDI
Par exemple, si vous indiquez en section
    [REPERTOIRES]
            CACHE=c:\Bcdiweb\ProgWeb\TravDemo\Cache
                       vous devez déclarer en section
    [MODELES]
                       la ligne
           CACHE=TravDemo/Cache

5.      ELECTRE

Si vous êtes abonnés à MémoElectre ou MémoElectre Plus, vous pouvez paramétrer la ligne ELECTRE
ELECTRE=C:\BcdiServ\ProgWeb\electre (enlevez le point-virgule en début de ligne pour activer cette option).
De quoi s'agit-il ? Si l'établissement dispose dun solde positif dunités pour le service MémoElectre ou est abonné à MémoElectre Plus, BCDI Web permet l'affichage automatique des imagettes de la 1ère de couverture, du sommaire, dune biographie ou de la 4ème de couverture des livres disponibles pris en ligne sur la Base Electre. Cet affichage est activé à partir de la reconnaissance des EAN (codes-barres) dune base documentaire sous BCDI.

Attention ! Vous ne devez pas modifier le nom de ce répertoire qui doit OBLIGATOIREMENT se trouver en sous-répertoire du répertoire dinstallation de Bcdi web.

3.      Section [LICENCE]

ORGANISME= (tapez ici EXACTEMENT la chaîne de caractères indiquée dans votre Licence Utilisateur en face de Nom Organisme)
CODE=(tapez ici EXACTEMENT la chaîne de caractères indiquée dans votre Licence Utilisateur en face de Code dinstallation)

Ces codes d'installation sont les mêmes que ceux de votre BCDI abonnement. Vous pouvez les retrouver sur votre Espace Client (disponible à partir du "?" du menu général de BCDI).

4.      Section [PARAMETRAGES APPLICATION]

1.       MARGUERITE

Si vous indiquez Marguerite=OUI, une icône dune couleur correspondant au début de la cote saffichera dans les résultats.

2.      PURGE

En minutes, le temps au bout duquel la « mémoire » dun client est libérée (conservation de la suite de sa recherche, progression de page en page).
(par défaut =10)

3.       RESEAU

Par défaut, le paramètre RESEAU=OUI est pris en compte (même si cette ligne est absente du fichier INI.

La ligne RÉSEAU doit contenir une valeur parmi les deux suivantes :

·         NON (si vous interrogez une copie de la base, il est alors possible de mettre la base à jour par simple copie dune nouvelle base venant écraser les fichiers de la précédente)

·         OUI, si vous interrogez la base réelle gérée par BCDI réseau en direct.

4.      BASEBIBLIO

Si vous indiquez OUI, dans les résultats renvoyés par BCDI Web, il ne sera plus fait mention des exemplaires (ni, bien sûr du compte et des réservations).

5.      COMPTE

Oui si vous voulez que les usagers voient leur compte-lecteur (longlet Compte). Si Reservations=Non, le compte est visible, mais il nest pas possible de supprimer des réservations.

6.      RESERVATIONS

Oui si vous voulez que vos usagers puissent réserver « à la volée ». Cette fonction est active, même si vous mettez NON en compte, et le compte sera visible (il nest pas pensable davoir le droit de réserver sans pouvoir éventuellement supprimer une réservation).

7.      JOURNAL

Si vous indiquez OUI, un journal des recherches au format Csv sera créé dans le répertoire de travail, portant le même nom que ler fichier INI, mais avec lextension .LOG.

8.       CACHE

(défaut = OUI)

9.        BLOC

La ligne BLOC doit contenir le nombre maximum de réponses dans une page de liste de résultats. Par défaut, ce nombre est de 10. Si le nombre est 0, il ny a pas de pagination dans la liste des réponses. Elles sont toutes dans une seule et même page de résultats.

10.  CONNECTIONS

La ligne CONNECTIONS contient le nombre maximum de connexions simultanées gérées par lapplication.

11.   REPONSES

La ligne REPONSES, si elle n, indique le nombre maximum de réponses renvoyées si laffichage des réponses se fait en une seule page (BLOCS=0) ou Liste Biblio en recherche expert (par défaut, le nombre maximum est fixé à 200). Surtout utile pour les résultats au format XML (empêche un utilisateur de décharger toute la base).

Nota : si la liste des réponses contient plus de 100 réponses, le format « Tout au format bibliographique » nest plus proposé.

12.  TIME OUT

La ligne TIME OUT, si elle existe, indique le temps, en minutes, au-dela duquel le programme renvoie un message derreur au navigateur si la base interrogée na pas répondu.

13.  CACHECONNECTIONS

La ligne CACHECONNECTIONS, si elle existe, indique si les connexions sont détruites après utilisations et renouvelées pour chaque nouveau client (NON), ou si elles sont conservées cachées en attendant dêtre réutilisées pour un nouveau client (OUI). Par défaut, cest la deuxième option (OUI) qui fonctionne.

14.    DEBUG

La ligne DEBUG indique éventuellement le nom dun fichier de log pour le débogage de lapplication (après les essais dinstallation, il est inutile de conserver ce fichier, placez un point-virgule devant cette ligne).

15.    XML

Par défaut, XML=NON. Autorise ou interdit le retour des résultats au format XML (voir ci-dessous le chapitre concernant XML).

16.  RAFRAICHISSEMENT ELECTRE

Par défaut = 14. Cette option, si elle est activée, permet de créer une base de données dans le répertoire ELECTRE, dans laquelle sont stockés les codes à barres des références déjà demandées à Electre.
Cette option évite de se reconnecter à Electre en affichant directement les affichettes ou textes stockés lors dune précédente visite dans le répertoire ELECTRE. A terme, ce fonctionnement rend BCDI web beaucoup plus rapide à laffichage.

17.    INTERDIT

Permet de paramétrer un complément déquation de recherche SAUF. Il est ainsi possible de rendre invisible à la recherche des notices, par exemple :
INTERDIT=TEMPORAIRE N.=OUI-
correspond à léquation de recherche « SAUF Temporaire N.=~oui-~ ».

18.    FILTRE

Permet de paramétrer un complément déquation de recherche ET. Il est ainsi possible de ne rendre visible quune partie des notices, par exemple :
FILTRE=TEMPORAIRE N.=NON
correspond à léquation de recherche « ET Temporaire N.=~non~ », cest à dire que seules les notices « définitives » dune base de travail par exemple seront vues.

5.       Section [AFFICHAGE]

PAGE=15 indique le nombre de pages à afficher dans la barre de déplacement dans les résultats

PRECEDENT=2 indique le nombre de pages que lon remonte lorsque lon clique sur précédent dans la barre de défilement des résultats.

6.      Section [MODELES]

Elle doit définir le nom des fichiers HTM qui servent de modèles pour linterface WEB. Il y en a trois, désignés dans les 4 lignes de cette section.

1.       FORMULAIRE

FORMULAIRE : le modèle de page daccueil pour la recherche. Par défaut, Index.htm (dans linstallation « standard », Index.htm est la page de recherche

2.       LISTE

LISTE : le modèle de page pour la liste des notices résultant de la recherche. Par défaut, Liste.htm (dans la présentation « standard » de BCDI Web, cette page sappelle ListeNot.htm)

RAPPORT RESERVATION=CONFIRM.HTM Affiche le fichier Confirm.htm pour indiquer à lusager son rang de réservation et en fonction de ce rang, une éventuelle confirmation. Si on commente cette ligne, aucune confirmation nest affichée et la réservation est effectuée directement.

Rappel (voir ci-dessus la section [REPERTOIRES])
Si vous modifiez le nom du répertoire cache, par exemple pour créer un répertoire cache spécifique à chaque base si vous installez plusieurs bases (ce répertoire cache doit OBLIGATOIREMENT avoir été créé par vous-même auparavant et se trouver en sous-répertoire du répertoire dinstallation de Bcdi web), vous devez AUSSI créer une ligne supplémentaire dans la section [MODELES] avec la syntaxe suivante :
CACHE=chemin html depuis le répertoire dinstallation de BCDI
Par exemple, si vous indiquez en section
    [REPERTOIRES]
            CACHE=c:\Bcdiweb\ProgWeb\TravaDemo\Cache
                       vous devez déclarer en section
    [MODELES]
                       la ligne
           CACHE=TravDemo/Cache

3.       Balises et conventions dans les modèles

Les modèles proposés peuvent être modifiés (avec prudence et ce nest vraiment pas simple, à réserver aux spécialistes Html !) pour être personnalisés, mais à condition de respecter lexistant (ou des règles strictes) concernant les balises.

Les balises sont placées dans les modèles HTML et sont interprétées par BCDI Web. Elles sont de la forme <#nom-de-la-balise> .

ATTENTION : A moins de modifier ces modèles avec un simple éditeur de texte, il faut vérifier que des éditeurs spécialisés HTML en mode graphique ne vont pas modifier la signalisation de ces balises en remplaçant notamment le crochet ouvrant < par sa valeur interprétée  &lt;  et le crochet fermant > par &gt;

En installation standard, le fichier ListeNot .htm sert de modèle pour laffichage des résultats en liste.
Ce fichier (ou votre propre fichier) DOIT se trouver dans le sous répertoire MODELES déclaré dans le fichier INI de lapplication.
Les balises devront correspondre à la nomenclature de BCDI pour les fichiers modèles HTM.
Voir les pages du site Club-BCDI :

http://club-bcdi.crdp-poitiers.cndp.fr/index.php?rub=nom_balise_modele

4.       Recherche directement depuis lURL

1.       Equation en ligne

On peut lancer une recherche directement dans lURL de départ. Pour cela, il faut, dans le navigateur internet, donner ladresse de lapplication :
http://MonServeur/MonAlias/bcdiweb.cgi/demo?
suivi dune requête du type :
?eq:[champ1=valeur1]& [champ1=valeur1]& [champ1=valeur1]
où champn sont des noms de champs et valeurn des valeurs recherchées.
On peut faire suivre les valeurs dun tiret pour une extension à droite.

Exemple :
?eq:(auteurs=perr-%20ou%20auteurs=grim-)&editeurs=gal-
devrait donner, entre autres, les livres de Perrault ou de Grimm édités par Gallimard.

 Notez bien que  :
- la requête commence par eq : (eq suivi des deux points) ;
- on peut mettre indifféremment des majuscules ou des minuscules ;
- on peut également ne mettre quun mot après le signe eq: Dans ce cas, ce mot sera recherché en plein texte ;
- les espaces doivent être remplacés par  %20 ;
- le nom des champs doit être EXACTEMENT celui qui apparaît dans BCDI ;
- vous pouvez utiliser les opérateurs booléens ET, OU et SAUF (mais en les encadrant de %20) et en pensant bien éventuellement aux nécessaires parenthèses.

Quelques exemples :

- vous pouvez, sur une page Html, créer un lien pour afficher dynamiquement vos nouveautés (ici, les données saisies depuis le premier janvier 2004) :
?eq:date%20de%20saisie>01/01/2007

- ou créer un lien pour afficher dynamiquement une bibliographie thématique :
?eq :(intranet%20OU%20internet)%20ET%20sécurité

Astuce :
Il peut être assez complexe décrire à la main une équation de recherchesurtout avec des parenthèses Il est beaucoup plus simple dutiliser BCDI en mode expert, de rédiger léquation de recherche, de la vérifier, puis, par copier-coller de lajouter au bout de ?eq : puis de remplacer les espaces par %20.

- vous pouvez passer en paramètres lutilisation de Dico (« tremblement de terre » permettra de trouver « séisme » entre autres) et le classement par pertinence :
?eq:tremblement%20de%20terre&langage=DICO&classement=PERTINENCE

- vous pouvez demander un classement sur tous les champs indexés de BCDI (sachant que le tri sur les dates est TOUJOURS inverse) : 
?eq:date%20de%20saisie>01/01/2004&classement=date%20de%20saisie

- vous pouvez demander lédition dans un format normalisé (BIBLIO ou ISBD) :
?eq:date%20de%20saisie>01/01/2004&classement=date%20de%20saisie&format=biblio

2.      Résultats au format  FORMAT XML :

Si, dans le fichier INI, XML=oui, il est possible depuis un navigateur (ou par programmation PHP, ASP ou autre) de demander à lapplication les résultats au format XML :
?eq:date%20de%20saisie>01/01/2004&classement=date%20de%20saisie&format=XML
Par défaut, le format XML renvoyé est le format MémoNotices XML complet de BCDI (voir dans laide de BCDI, sommaire, annexe, DTD).
Il est aussi possible de proposer un fichier modèle dextraction.
?eq:France&format=XML&fichier= FICHE_EXE_XML.TXT
 Pour cela, le fichier correspondant à LISTE devra lui-même être un fichier XML, contenant les balises de votre choix, et, entre ces balises, le nom des champs avec la syntaxe BCDI (cf modèles HTM ci-dessus).
Attention ! Ce fichier DOIT être placé dans le sous-répertoire MODELES.
Dans ce fichier modèle, vous pouvez insérer un autre fichier XML (par exemple dans FICHE_EXE_XML.TXT) qui bouclera pour afficher les champs de tous les exemplaires (<#LISTE_EXEMPLAIRES FICHE_EXE_XML.TXT>

Exemple du fichier XML.TXT livré dans MODELES :

<?xml version="1.0" encoding="ISO-8859-1" ?>
<MEMO_NOTICES count="<#nb_reponses>" session="<#connexion>">
<NOTICES>
<TYPE_NOTICE_N><#Champ Notices Type Notice></TYPE_NOTICE_N>
<RESUME_N><#Champ Notices Résumé></RESUME_N>
<DESCRIPTEURS_N><#Champ Notices Descripteurs></DESCRIPTEURS_N>
<MOTS_CLES_N><#Champ Notices Mots clés></MOTS_CLES_N>

5.      Le journal des recherches

Les requêtes envoyées à BCDI Web par les utilisateurs sont enregistrées dans un fichier du même nom que le fichier de paramétrage avec lextension .log (par exemple BcdiWeb_Data.log). Ce fichier est automatiquement créé au lancement de lapplication dans le répertoire de travail indiqué dans le fichier INI..
Ce fichier peut être relu depuis tout logiciel capable de relire des fichiers CSV (séparateurs « ; »)

Conseil : recopiez (ou mettez en téléchargement) le fichier journal sur la station de travail (Windows) du (ou des) responsable(s) de la base de données en leur indiquant ladresse de téléchargement du fichier journal :
http://NomDuServeur/AliasBcdiWeb/TravXxx/bcdiweb_Xxx.log
ils pourront ainsi enregistrer ce fichier localement pour le lire et linterpréter..

Prévoyez de stocker puis détruire « périodiquement » (à décider avec le (les) responsable(s) de la base) ce fichier pour éviter de le voir augmenter démesurément

 

 

6.      Intégration dans les ENT 

Démonstration de ce qui est possible avec BCDI Web à travers des exemples en ligne.

Dans ce qui suit, nous sous-entendons BCDI web installé (consulter les documents dinstallation de BCDI Web) sur un serveur Windows ou un serveur linux accessible via internet à lensemble des usagers.

Une BCDI web Démo en ligne a été installé pour effectuer tous les essais

 http://doc-partage.crdp-poitiers.cndp.fr/betatest/bcdiweb.cgi/demo

Nota : dans cette base BCDI exemple, lidentifiant et le mot de passe sont identiques, choisir un n° inférieur à 200.

 

Identification

Si dans sa version actuelle, BCDI Web nest pas compatible SSO, il est possible de passer une identité et un mot de passe en paramètres à lURL.

Il serait donc possible, pour éviter à un usager de lENT de se réidentifier dans BCDI Web à chaque visite :

- soit dimporter dans BCDI lidentité de lENT (« dangereux » au niveau de la confidentialité

- soit dimporter dans lENT lidentité de BCDI (ce qui nécessite des mises à jours au moins annuelles)

- soit de lui demander dans lENT dindiquer son identité et son mot de passe dans BCDI la première fois, lENT stockant ces informations pour éviter à avoir à les redemander  (lENT vérifiant automatiquement à chaque session que ces renseignements sont toujours exacts). Cest cette solution que nous préconisons.

1.       Pour vérifier si lidentification est correcte (passage de paramètres dans lURL) :

Exemple :

http://doc-partage.crdp-poitiers.cndp.fr/betatest/bcdiweb.cgi/demo?IDENTIFIANT=1&MOT_DE_PASSE=1

BCDI répond par un fichier XML dont le modèle (paramétrable) peut être

<?xml version="1.0" encoding="ISO-8859-1" ?>

<ID><#VALEUR IDENTIFICATION></ID>

Dans lequel <#VALEUR IDENTIFICATION> prend la valeur :

·         1 si l'identification est correcte dans la base interrogée et est autorisée par le fichier INI de Bcdi web

·         0 si les paramètres d'appel ne contiennent pas IDENTIFIANT ou MOT_DE_PASSE 

·         -1 si l'identification est correcte dans la base interrogée mais quelle n'est pas autorisée par le fichier INI de BCDI web,

·         -2 si l'IDENTIFIANT n'existe pas dans la base interrogée

·         -3 si le MOT_DE_PASSE ne correspond pas à l'identifiant dans la base interrogée

Par exemple :
http://doc-partage.crdp-poitiers.cndp.fr/betatest/bcdiweb.cgi/demo?IDENTIFIANT=2&MOT_DE_PASSE=2&fichier=id.xml
répond
<ID>1</ID>
http://doc-partage.crdp-poitiers.cndp.fr/betatest/bcdiweb.cgi/demo?IDENTIFIANT=3&MOT_DE_PASSE=2&fichier=id.xml
répond
<ID>-3</ID>

Nota : dans quels cas l'identification est-elle non autorisée dans le fichier INI de BCDI Web ?

- si dans le INI de BCDI Web BASEBIBLIO=OUI (prioritaire)

- si dans le INI de BCDI Web COMPTE=NON ET RESERVATIONS=NON (si RESERVATIONS=OUI, COMPTE est forcé à OUI)

On peut très bien avoir COMPTE=OUI et RESERVATIONS=NON, ce qui est réglé dans l'affichage des résultats (ListeNot.htm)

On voit l'onglet compte mais pas le bouton Réserver.

Extrait du fichier INI (installation par défaut) :

[PARAMETRAGES APPLICATION]

BASEBIBLIO=NON

COMPTE=OUI

RESERVATIONS=OUI

REPONSES=200

XML=NON

Nota :

Par défaut, loption XML=NON, vous devez donc modifier ce paramètre en mettant XML=OUI pour obtenir une réponse XML (ce qui ninterdit pas une réponse Html si vous interrogez BCDI sans passage de paramètres à lURL ou sans indiquer formellement &Format=XML).

 

Dans les exemples ci-dessous, nous estimons que lidentification est correcte (elle a été testée auparavant).

7.       Paramétrer à distance les écrans de recherche

Vous pouvez passer un paramètre à lURL dinterrogation de la base :

?EDITER_VAR=XXXXXX (où XXXXXXXX est le code dinstallation de BCDI tel quil apparaît dans le fichier INI

Vous obtenez un écran du type :


Il est alors possible de cocher / décocher les options daffichage.

Lorsque vous enregistrez, les modifications sécrivent sur le serveur et vous retrouvez lécran de recherche de BCDI web réinitialisé.

Note technique pour les spécialistes :
Il sagit ici de déclarer des variables (de fait, des constantes) qui serviront plus loin dans les écrans de recherche et/ou de restitution grâce à une syntaxe très simple :
<#SI AFFDISCIPLINES=OUI>
on écrit ce qui doit être fait
<#SINON AFFDISCIPLINES=OUI>
On écrit ce qui doit être fait si la première condition nest pas remplie
<#FINSI AFFDISCIPLINES=OUI>
par exemple (pris dans Restrictions.htm des modèles collège) :
<td width="120">
     <#SI AFFDISCIPLINES=OUI>
           &nbsp;&nbsp;<img src="<#CONST REP_MODELES>puceoptions.gif" width="10" height="12" alt="">&nbsp;
            <b>Disciplines : &nbsp;&nbsp;&nbsp;&nbsp;</b>
     <#SINON AFFDISCIPLINES=OUI>
            &nbsp;
      <#FINSI AFFDISCIPLINES=OUI>
</td>
<td width="200">
      <#SI AFFDISCIPLINES=OUI>
          <select id="DISCIPLINES" name="DISCIPLINES">
          <option selected>Tous</option><#OPTION_DISCIPLINES largeur=40>
         </select>
       <#SINON AFFDISCIPLINES=OUI>
           &nbsp;
       <#FINSI AFFDISCIPLINES=OUI>
  </td>
ce qui offre bien des possibilités, sachant que vous pouvez déclarer les constantes de votre choix
Vous pouvez aussi utiliser ces constantes pour tout autre chose, par exemple, vous déclarez :
<#VAR CONTACT=LeMelDuResponsable@Prestataire.fr>
et vous écrivez tout simplement dans le code des pages, là où vous le  jugez utile
<#CONTACT>
BCDI remplacera partout ce tag par son contenu : le jour où le responsable changera, il vous suffira de modifier la déclaration de la constante dans le fichier Index.htm (déclarez vos constantes dans le fichier Index.htm, car cest lui est est exécuté le premier) pour que toutes vos pages prennent en compte cette modification.
Remarque : Vous pouvez aussi déclarer des constantes dans le fichier INI de BCDI Web, mais elles ne seront actives que pour BCDI Web, alors que les modèles Html sont strictement identique en BCDI Web et en interface Html intégrée à BCDI client-serveur, donc des constantes déclarées dans index.htm seront communes.
